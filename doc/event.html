事件

事件可以将用户代码注入到请求生命周期各个阶段触发执行。

事件类初始化时默认读取event配置项，并注册配置事件。

基本使用

优先级

同一事件标签下注册了多个优先级相同执行方法时，系统不能保证方法按注册先后顺序执行。

中断执行

在同一事件标签注册了多个执行方法时，多个方法会按优先级依次执行，如果需要其中方法执行后中断后续其他方法执行，则只需此方法返回false。

内置事件

框架已内置7个事件标签，这些事件将在请求生命周期的不同阶段触发。

  • boot

    触发在应用初始化App::boot()后，可以用于添加一些环境初始化代码，如时区设置等。

  • start

    触发在应用实例化App::start()后，可以用于添加一些dispatch功能代码，例如权限验证，CSRF。

  • request

    触发在获取request时，可以用于初始化一些request信息，例如trim所有POST数据的空白字符。

  • exit

    触发在发送应用结束阶段，也就是register_shutdown_function()后，在这阶段框架会释放一些资源，优化内存使用。

  • flush

    触发在exit事件之后，用于输出视图页面或序列化的数据（默认在Response类初始化时注册）。

  • response

    默认在执行flush事件方法时触发，用于修改response数据，比如添加一个公共的response header头，或压缩数据等。

  • close

    触发在连接断开后，也就是执行fastcgi_finish_request()后，在此阶段客户端已经接收到了响应并断开了连接，可以将一些比较耗时的操作如 发送邮件 日志处理 等放在此阶段执行。
